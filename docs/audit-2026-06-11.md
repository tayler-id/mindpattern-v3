# Full-System Audit — 2026-06-11

Six parallel deep reviews (orchestrator, memory/data layer, social subsystem, Slack bot + dashboard,
harness + knowledge graph, security) plus a test-suite baseline. Every finding below was verified
against the actual code by the reviewing agent before being reported.

Test baseline: `.venv/bin/python3 -m pytest tests/ -q` → **7 failed, 892 passed**
(test_compile ×2, test_cors ×2, test_mirror archival, test_sync extraction-fallback, test_trend_history).

## CRITICAL

### C1. Entire `data/` tree served unauthenticated — `dashboard/app.py:54-56`
`app.mount("/data", StaticFiles(directory=DATA_DIR))` with no auth. `curl https://mindpattern.fly.dev/data/ramsay/memory.db`
downloads the full personal database; identity files (`soul.md`, `user.md`, `voice.md`) and drafts too.
**Fix:** remove the broad mount; serve only an art/assets subdirectory or gate behind auth.

### C2. No global auth on the dashboard — `dashboard/auth.py:34-46`, `dashboard/app.py`
`require_auth`/`is_public_route`/`PUBLIC_PREFIXES` exist but are wired into only 4 routes in `api.py`.
Unauthenticated state-changing endpoints include:
- `POST /api/editors-desk/{id}/approve` → live posts to Bluesky/LinkedIn/X (`editors_desk.py:503-535`)
- `POST /api/engagement/post` → posts replies + auto-follows (`editors_desk.py:796`)
- `POST /prompts/{agent}/revert` → runs `git checkout`/`git commit` on the server repo (`prompts.py:290-325`)
- `POST /api/approvals` / `GET /api/approvals/{token}` → approval-queue poisoning (`api.py:832,924`)
**Fix:** default-deny middleware using the already-written `is_public_route`.

### C3. Personal database committed to git
`data/ramsay/memory.db.backup-small` (1.6 MB, real findings/feedback/engagements/preferences) and
`data/memory.db` are tracked. `.gitignore` covers `data/*/memory.db` but not the backup suffix.
`social-config.json` is ALSO tracked (contains phone number; CLAUDE.md says never commit) and is
COPY'd into the Docker image; `users.json` (real emails) likewise.
**Fix:** `git rm --cached`, purge history (filter-repo/BFG), broaden `.gitignore`, load config from secrets.

### C4. Approval gate defaults to "post everything" — `social/approval.py:369-389, 577-580`
The poll loop accepts ANY human message in #mindpattern-approvals after the gate message (fallback is
not thread-scoped), and `_parse_draft_reply` defaults to `action: "all"` for any reply not starting
with n/s/k/c. "wait" or "hold on" publishes to every platform. No approver-identity check
(`approval.py:361,386`). Approver sees only first 500 chars of drafts / 150 chars of replies.
**Fix:** explicit affirmative tokens only; thread-scope the fallback; allowlist approver user IDs; show full text.

### C5. Failed posts recorded as published — `social/pipeline.py:781-806, 660-692`
Platform clients return `{"success": False}` instead of raising (`posting.py:602-607, 1169-1174`),
but `_post_with_jitter`/`_post_pending` never check the flag → `store_post(posted=True)`,
`mark_pending_posted`, Slack reports success. Expired LinkedIn token (no expiry check in pipeline path,
`posting.py:905-913`) silently eats every post while consuming rate limit.
**Fix:** check `post_result["success"]`; verify LinkedIn token at pipeline start and alert on 401.

### C6. Research agents hold `Bash` while reading scraped content — `orchestrator/agents.py:52-54`
`AGENT_ALLOWED_TOOLS = ["Agent","Bash","WebSearch","WebFetch",...]` pre-approved, prompts embed
untrusted scraped text and instruct fetching arbitrary URLs (`agents.py:223`). A crafted page can
drive shell execution on the Mac that holds Keychain secrets and git credentials. The injection
"scanner" (`policies/engine.py:445-447`) is an 11-phrase substring match run AFTER execution, on output only.
**Fix:** drop `Bash` (and `Agent`) from research-agent allowed tools; same for harness research agent's `WebFetch`.

### C7. Engagement replies bypass ALL policy validation — `social/engagement.py` (no call site)
LLM-drafted replies to strangers are never run through `validate_social_post` (the broadcast pipeline
validates 3×; the highest-risk path validates 0×). No banned words/entities, no grapheme limit, no
injection screening of the stranger's post text (`engagement.py:946-960`). Auto-follow result never checked.
**Fix:** validate every `our_reply` before posting; `scan_for_injection` candidate text; check follow results.

### C8. Harness worktree misattribution under parallelism — `harness/run.sh:183, 116-133`
`find_newest_worktree` returns the single newest worktree; with `PARALLEL_AGENTS=3`, slots can grab
each other's worktrees — gates/scope checks run against the wrong ticket's diff, wrong PR opened.
**Fix:** slot-specific worktree paths passed explicitly.

### C9. Identity files: destructive replace, no snapshot — `memory/identity_evolve.py:323-351`
`apply_evolution_diff` full-section-replaces `soul.md`/`voice.md` with LLM output. No git snapshot,
no version record, no length/similarity guard, never auto-committed. One hallucinated diff wipes
accumulated identity. `_UNSAFE_PATTERNS` rejection guard is dead code (`identity_evolve.py:39-44`).
**Fix:** snapshot before apply; reject updates <50% of existing section length; wire or delete `_UNSAFE_PATTERNS`.

## HIGH

### Orchestrator
- **DELIVER not idempotent on resume** — newsletter can double-send to user + all subscribers
  (`runner.py:855-914`, checkpoint written after handler returns at `runner.py:137`). Record a
  per-date send receipt.
- **`run_claude_prompt` lacks process-group kill** (`agents.py:942-950`) — the exact stall/orphan bug
  already fixed in `run_single_agent` (`agents.py:392-403`); this is the synthesis path with the
  documented opus stall history. Apply the same Popen/killpg pattern.
- **Unvalidated LLM JSON kills the run** — `_parse_findings` (`agents.py:514,520`) returns
  `data["findings"]` unchecked; `{"findings": null}` → TypeError at `runner.py:498` → RESEARCH is
  critical → whole pipeline fails.

### Data layer / sync
- **Split-brain by design** — sync is push-only (`sync.py:41-196`); Fly-side writes (e.g. `#mp-posts`
  publishes — never recorded in memory.db at all, `slack_bot/handlers/posts.py:304-358`) are clobbered
  daily. Dedup and daily post-caps are blind to bot-published posts.
- **Stale-WAL corruption hazard** — tar replaces `memory.db` under live Fly processes, old `-wal`/`-shm`
  remain and get replayed against the new file (`sync.py:151`; `restart_app()` exists at `sync.py:291`
  but `_phase_sync` never calls it). Bundle is a raw file copy of a live DB, not `.backup`/`VACUUM INTO`
  (`sync.py:230`), and checkpoint failure is ignored (`sync.py:80-82`).

### Slack bot
- **Fails open** — no owner ID configured → bot obeys any workspace member (`bot.py:64-68, 156-160`).
- **Mutex is dead code** — `_acquire_mutex` never called; Mac+Fly bots double-process every event
  (`bot.py:71-81`); no event dedup by `event_ts` (`bot.py:127-145`).
- **`"skip linkedin"` posts to LinkedIn** — substring approval parsing (`handlers/posts.py:287-302`);
  engagement sniper posts any non-keyword text as the literal reply.
- **Handlers block the Socket Mode worker pool** for up to 15 min (`bot.py:168-170`, `handlers/base.py:78-101`).

### Harness
- **Interpreter split** — gates use `$VENV_PYTHON` but all ticket-lifecycle calls use bare `python3`
  (3.9.6, crashes on `dict | None`), mostly `2>/dev/null || true` → the loop silently no-ops
  (`run.sh:153-362`; `tickets.py:28`). This is also why `python3 -m pytest` fails at the repo root.
- **Tickets stranded `in_progress` forever** — no recovery path (`run.sh:325-356`, `tickets.py:131-159`).
- **Review threshold never enforced in code** — `review_pass_threshold: 7` is read by nothing; PR
  creation is up to the LLM following prose (`run.sh:223-261`, `prompts/review.md:79-93`). Human merge
  + CI is the only real gate (no `gh pr merge` anywhere — good).
- **`knowledge check` can never pass** — `scan_code_refs` matches `@know` examples in its own
  docstrings (`knowledge_sections.py:500-552`); 4-pass validation is permanently red, hook spews
  false warnings.

### Social/platform
- **Bluesky mention facets send raw handle as `did`** (`posting.py:485-496`) → AT lexicon rejects any
  post/reply containing `@user.tld` — engagement replies are the most likely to mention.
- **`count_graphemes` skips `Sk` chars** (backticks!) (`policies/engine.py:20-30`) → over-limit
  technical posts pass policy, rejected by PDS, recorded as posted (per C5).
- **Reply cooldown is platform-wide, not per-author** (`engine.py:360-379`) → ~1 reply/week total;
  meanwhile `max_replies_per_day` never checked against DB.
- **Non-idempotent publish endpoints retried** on timeout/5xx → duplicate live posts (`posting.py:87-127`).

## Design assessment (cross-cutting)

1. **Safety code is written but not wired — the defining pattern of this codebase.** Auth middleware
   helpers, the bot mutex, the owner check (fails open), `_UNSAFE_PATTERNS`, `review_pass_threshold`,
   `Checkpoint.load()`, `restart_app()`, `cleanup_stale_runs`, cost logging, archival — all exist as
   functions and all have zero callers or fail-open wiring. The system looks far more gated than it is.

2. **The human approval gate — the one real safety mechanism — is the most permissive code in the repo.**
   Any channel message counts, ambiguity defaults to "post everything", content is truncated, approver
   identity unchecked, and the dashboard exposes an unauthenticated bypass of the same gate.

3. **Replication is push-only file-clobbering of a live WAL database.** No pull path, no read-only
   enforcement on Fly, stale-WAL corruption vector, bot-published posts invisible to dedup/caps.

4. **Checkpointing without idempotency or state restoration.** Resume is a different (degraded)
   execution mode; the only phases with irreversible external effects (DELIVER/SOCIAL) have no receipts.

5. **Three overlapping observability layers** with conflicting schemas for the same tables; half the
   schema written by nobody or read by queries that can never succeed.

6. **Effort is inverted: over-engineered where it's safe, under-engineered where it's dangerous.**
   1,360 lines of knowledge-graph machinery (5-tier fuzzy search, 4-pass validation that can never pass)
   vs. zero deterministic enforcement on the paths that post publicly, execute shell, and rewrite identity.

## Recommended rebuild priorities

1. **Lock the perimeter (hours):** default-deny auth middleware; kill the `/data` mount; purge
   DB/PII from git history; fail-closed bot owner check.
2. **Fix the action boundary (days):** success-flag checks + send/post receipts (idempotency keys
   keyed on run-date+platform); strict approval parsing (explicit allowlist of approvers + affirmative
   tokens, thread-scoped); validate engagement replies; drop Bash/WebFetch from agents reading
   untrusted content; single env-var kill switch that hard-disables all outbound writes.
3. **Fix the data spine (days):** one schema owner + `PRAGMA user_version` migrations; `Connection.backup()`
   for sync bundles; make Fly copy read-only and route Fly-side writes through a queue (or move the
   write-master to Fly and have the Mac push/pull via API); standardize timestamps.
4. **Simplify (ongoing):** delete the dead third of the codebase (claims, cost plumbing, dead phases,
   `_web_approval`, deferral half-feature); collapse observability to one layer; shrink the knowledge
   graph to link-resolution + keyword search; make `runner.py` phases take explicit inputs/outputs
   (the checkpoint `state_data` was the intended vehicle) so resume semantics become per-phase contracts:
   recompute / DB-derived / receipt-guarded.

## Addendum (2026-06-11, evening): the patterns table has been dead since March 11

Discovered while studying the corpus, not part of the original audit sweep:

- All 97 rows in `patterns` have `last_seen <= 2026-03-11`; zero pattern activity in 3 months
  while findings (~130/day) and newsletters kept flowing.
- Root cause: the v2→v3 rewrite (initial commit 2026-03-14) ported `memory/patterns.py` —
  improved and fully tested — but **nothing in the production pipeline ever called
  `record_pattern()`**. The runner only called the maintenance functions (`consolidate`,
  `promote`), which groom existing rows. The organ was transplanted but never reconnected.
- Significance: this is the strongest single piece of evidence for the spec's M1 thesis — the
  long-term synthesis layer silently died and nobody noticed for a quarter. The temporal KG
  (M1, with full backfill of all findings) replaces this mechanism outright.
- Disposition: `record_pattern`/`get_recurring` deleted as orphans in M0 task 17
  ("functions that exist must have callers"); `consolidate`/`promote`/`prune`/`store_note`
  remain (still called). The frozen `patterns` table stays in place — `/api/patterns` is part
  of the pinned Vercel contract — until M1 supersedes it.
