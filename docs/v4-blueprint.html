Part 1

What we're building & why

Today, MindPattern is a newspaper: every morning it researches, writes, publishes — and forgets. v4 turns it into a library that compounds: every finding becomes a dated fact in a knowledge graph, every one of your reactions becomes training signal, and once the library is rich enough, an agent mines it to answer the only question that matters: “with everything you know, what should I build?”

📰 v3 — the newspaper (today)

  • 13 agents research → newsletter → posted → forgotten
  • “Knowledge graph” = hand-maintained markdown links → noise
  • “Self-improvement” = LLM rewrites its own prompts blind → no measurable gain (the literature now calls this context collapse)
  • Yesterday's research barely helps today's

📚 v4 — the compounding library

  • Same daily run, but every finding becomes typed, dated facts about entities
  • Facts get superseded, never deleted → story timelines
  • Improvement only ships if it beats the current version on replays of real past work
  • Month 6 is categorically smarter than week 1 → opportunity engine

Part 3

Five mental models

Each new piece of v4 maps to something familiar. If a model feels wrong, comment on it — these drive the design.

🗂️

The knowledge graph is a card catalog with dated cards

Every fact is a card: “OpenAI —released→ GPT-5” with a valid-from date. When news changes, the old card gets stamped “superseded on date X by card Y” — never thrown away. Ask for any entity and you get its timeline, not a soup of contradictions. (This is exactly how Bloomberg's graph and Zep's Graphiti work.)

🚗

The replay gate is Tesla's shadow mode

Tesla never ships a new network directly — it rides along silently and gets compared against the current one on real driving. Our version: any new prompt/lesson runs against 50+ real past inputs (seeded from your actual edits) and only ships if it beats the incumbent. The industry numbers say ~90% of changes lose — that's why the gate exists.

📓

Agents learn via a coach's notebook, not brain surgery

v3 let the LLM rewrite its own prompt — Stanford measured that exact pattern collapsing (18k tokens → 122, worse than baseline). v4 instead appends numbered lessons: “#23: Tayler always cuts hype adjectives (helpful 7 / harmful 0)”. Lessons earn or lose points; useless ones retire. The playbook is never rewritten wholesale.

🎣

Source & topic selection works like fishing spots

Each source/topic is a spot. Catch something (you click, react, use a finding) → fish there more. Nothing for weeks → fish less, but never abandon a spot (the news shifts; old spots come back). This is Thompson sampling — the same math behind Netflix artwork and Spotify podcasts — and it's the one thing that genuinely learns online at 1 run/day.

⛏️

The opportunity engine is a prospector who must show his instruments

He doesn't dig wherever feels lucky — instruments (mention velocity, cross-source corroboration, gap clusters) say where. The LLM never invents a candidate; it only evaluates the spots the math surfaced, must dig up counter-evidence too, and every claim cites finding IDs you can audit. His verdicts get re-checked every week — over months you learn whether to trust him.

Part 5 · build first

The knowledge graph, concretely

Three layers, four daily stages

Layer 1 · Episodesyour findings table, untouched — raw provenance
→
Layer 2 · Entities & factstyped, deduped, dated edges
→
Layer 3 · Communitiesweekly topic clusters + summaries

Stage 1 — Extract (daily, ~15 claude calls): each finding → typed entities (Company, Product, Person, Tech, Paper, Event, Funding) and atomic facts using ~16 fixed verbs (RELEASED, RAISED, ACQUIRED, COMPETES_WITH, PREDICTS…). Typed, not open-ended — open extraction is the #1 documented source of graph noise.

Stage 2 — Resolve (the make-or-break stage): is “Anthropic PBC” the “Anthropic” we know? Alias table first (free), then embedding+keyword shortlist, then one LLM adjudication call. Never auto-merge on similarity alone (that's how “GPT-5” and “GPT-5-mini” become one blob). Uncertain merges get flagged for you — Apple's quarantine queue, miniature.

Stage 3 — Upsert & invalidate: new fact contradicts an old one between the same entities? Old edge gets invalid_at stamped + a pointer to what superseded it. Timeline preserved.

Stage 4 — Consolidate (weekly, ~10 claude calls): cluster the graph into topic communities, re-summarize only the changed ones, recompute importance (PageRank × mentions × recency). Demote, don't delete.

What a fact actually looks like

● VALID  OpenAI —RELEASED→ GPT-5.2
fact: "OpenAI released GPT-5.2 with 4× cheaper inference" · valid_at 2026-05-28 · episode #13201 (arXiv + HN) · confidence .95
● SUPERSEDED  OpenAI —FLAGSHIP_MODEL→ GPT-5
valid 2025-08-07 → 2026-05-28 · invalidated_by episode #13201 · still queryable: “show timeline of OpenAI flagships”

The schema — 5 tables, that's it

-- episodes = your existing findings table (13,894 rows ready to backfill)
CREATE TABLE entities (
  id INTEGER PRIMARY KEY, canonical_name TEXT, type TEXT,     -- 'company'|'product'|...
  summary TEXT, embedding BLOB, importance REAL DEFAULT 0,
  first_seen TEXT, last_seen TEXT, mention_count INTEGER DEFAULT 1);

CREATE TABLE entity_aliases (alias TEXT PRIMARY KEY, entity_id INTEGER);

CREATE TABLE edges (
  id INTEGER PRIMARY KEY, src INTEGER, predicate TEXT, dst INTEGER,
  fact_text TEXT, episode_id INTEGER,                -- provenance, always
  fact_kind TEXT,                                    -- 'fact'|'opinion'|'prediction'
  valid_at TEXT, invalid_at TEXT, invalidated_by INTEGER,
  confidence REAL, created_at TEXT);

CREATE TABLE communities (id INTEGER PRIMARY KEY, label TEXT, summary TEXT, updated_at TEXT);
CREATE TABLE community_members (community_id INTEGER, entity_id INTEGER);

No Neo4j, no vector database, no framework. At 15 findings/day the graph is a few thousand entities after a year — NetworkX holds it in memory for the weekly clustering, and brute-force cosine over SQLite blobs takes milliseconds. The discipline is in the pipeline, not the infrastructure — that's the verified lesson from Apple/LinkedIn/Airbnb, who all run relational tables too.

Part 6 · build second

The improvement harness

Honest framing first: at one newsletter/day you cannot A/B-test prompts online (you'd need ~2 months per comparison). So v4 learns online only over small choices (which sources/topics get attention — the bandit) and validates everything else offline against your archive (the replay gate). Your explicit feedback is the scarcest, most valuable signal — every edit you make becomes a permanent test case.

What a weekly opportunity looks like

PURSUE · score 21/30 — Compliance copilots for AI-agent audit trails

Why now (cites episode #12881, #13002): EU AI Act enforcement began May 2026; 3 YC W26 companies adjacent but none owns audit-trail UX.
Demand 4/5 — gap cluster: 17 complaint-findings about "can't explain what our agents did", zero linked products.
Timing 5/5 — ladder: arXiv+GitHub 9 months ago → HN front page 3× this month → first job postings (2 in HN-hiring).
Competition 3/5 — 4 adjacent entities, all velocity-flat.  Owner fit 4/5.
Counter-evidence (required): episode #11990 — incumbent observability players adding "agent logs"; risk this becomes a feature, not a product.
Watch trigger: re-raise if "agent audit" mention-velocity 2× or a funding event lands. Verdict re-scored weekly; track record visible.

The structural rule that makes this trustworthy: candidates come from the math, not the model. The LLM only interrogates entities/gaps the signals surfaced, counter-evidence retrieval is mandatory, and every score must cite finding IDs a validator mechanically checks. No citation → score is null, shown as "unknown" — calibrated ignorance over confident fabrication.

Those budgets are a contract: if a module wants to grow past ~2× its budget, that's a design smell we stop and discuss — not a milestone. Total new code ≈ 2,000 lines. For scale: v3's markdown knowledge-graph machinery alone is 1,360 lines, and it gets deleted.

Pattern 1 — every LLM call goes through one gate

# core/llm.py — the ONLY place subprocess meets claude. Fixes v3's three
# recurring bugs at once: orphaned processes, unvalidated JSON, silent failures.
def ask(prompt: str, schema: dict, *, model: str = "sonnet", timeout: int = 300) -> dict | None:
    """One claude -p call. Returns schema-valid dict, or None (logged). Never raises."""
    proc = subprocess.Popen([...], start_new_session=True)   # killpg on timeout — no orphans
    out = _communicate_or_kill(proc, timeout)
    for attempt in range(2):                                 # parse → validate → one retry
        data = _extract_json(out)
        if data and _matches(data, schema):
            return data
        out = _retry_with_error_feedback(prompt, out)
    log.warning("llm output invalid after retry: %s", prompt[:80])
    return None                                              # caller skips item; pipeline never crashes on bad JSON

Pattern 2 — entity resolution (the make-or-break 40 lines)

# kg/resolve.py — cheap → cheap → one LLM call. Every merge logged.
def resolve(db, name: str, etype: str, context: str) -> int:
    if (eid := db.alias_lookup(name)):                       # 1. exact alias — free, hits ~80% after warmup
        return eid
    cands = db.shortlist(name, etype, k=5)                   # 2. embedding-cosine ∪ FTS5 keyword, same type only
    if not cands:
        return db.create_entity(name, etype, context)
    verdict = llm.ask(ADJUDICATE.format(name, context, cands), MERGE_SCHEMA)
    if verdict is None or verdict["confidence"] < 0.8:
        eid = db.create_entity(name, etype, context)
        db.flag_for_review(eid, cands)                       # 3. uncertain → Apple-style quarantine, you skim weekly
        return eid
    if verdict["same"]:
        db.add_alias(name, verdict["entity_id"])
        db.log_merge(name, verdict)                          # audit trail — silent merges are how graphs rot
        return verdict["entity_id"]
    return db.create_entity(name, etype, context)

Pattern 3 — the replay gate

# harness/replay.py — Netflix-style paired comparison. ~90% of candidates SHOULD die here.
def gate(db, candidate_id: str) -> dict:
    cases = db.golden_cases(limit=60)                        # seeded from YOUR past edits + judge extremes
    wins = losses = 0
    for case in cases:
        a, b = run_both(case.input, incumbent, candidate)    # same input, both configs
        side = random.choice([(a,b), (b,a)])                 # judge is blind to which is which
        v = llm.ask(PAIRED_JUDGE.format(case.rubric, *side), VERDICT_SCHEMA)
        wins += v["winner"] == which(candidate, side)
        if case.source == "owner_edit" and v["winner"] != which(candidate, side):
            losses += 1                                       # regression on a case YOU corrected = special
    promote = wins / len(cases) >= 0.60 and losses == 0
    db.record_gate_run(candidate_id, wins, losses, promote)
    return {"promote": promote, "win_rate": wins/len(cases),
            "needs_eyeball": candidate.touches_style}         # style changes → you see a side-by-side first

How we keep it from getting over-engineered

  • No new infrastructure. SQLite + NetworkX + the claude CLI you already pay for. Zero servers, zero frameworks, one new pip dependency.
  • Plain functions, no class hierarchies. Every module is functions over a db connection — the style memory/claims.py already uses (the best code in v3).
  • Line budgets are enforced — growth past 2× budget triggers a design conversation, not a refactor sprint.
  • Delete as we go: v3's markdown-KG machinery (1,360 loc), claims, cost plumbing, dead phases, `_web_approval` — removed in the same milestones that replace them. v4 should land net smaller than v3.
  • Each milestone ships something you use that week — no big-bang integration at the end.
