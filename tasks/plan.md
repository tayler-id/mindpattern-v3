# Implementation Plan: Voice A/B + Prose-Craft Integration

> Source spec: `SPEC.md` (root)
> Generated by `/plan` on 2026-04-12
> Status: awaiting approval

## Overview

Two outcomes: (1) extract a "builder essayist" voice register from 16 donor writers and prove it beats current voice via a replay benchmark on the last 5 pipeline runs; (2) once promoted, capture every Slack-edited social post and feed the diffs into a daily learning loop. Reuse `social/approval.py` ApprovalGateway, `memory/db.py` schema-init pattern, and `orchestrator/agents.py` identity-loading hook. No new daemon code, no new pipeline phases.

## Architecture decisions (locked in spec)

- **Donor source**: blended corpus of 16 builder/founder essayists, not Tayler's writing.
- **Benchmark mode**: replay last 5 pipeline runs (variant A from disk, variant B regenerated against same inputs). Not a multi-week shadow, not curated topics.
- **Approval gate**: extend existing `ApprovalGateway` in `social/approval.py`. No new `slack_bot/` handler — bot daemon and orchestrator are separate processes; orchestrator calls ApprovalGateway in-process.
- **Newsletter is not gated.** Variant B applies to newsletter, but newsletter skips Slack approval and live edit-capture. Learning loop only learns from social writers.
- **Migrations**: add tables to `memory/db.py:_init_schema()` executescript. CREATE TABLE IF NOT EXISTS pattern. No separate migration runner.

## Critical gap discovered during planning

**Writer inputs are not stored for replay.** `findings` table in `memory.db` preserves newsletter inputs keyed by `run_date` (memory/db.py:63–74), so newsletter replay works. **But for LinkedIn and Bluesky writers, the brief/topic/source-URL trio that fed each writer is not persisted** — `social_posts.brief_json` column exists (memory/db.py:169) but pipeline.py never populates it. This blocks the replay benchmark for social writers.

**Mitigation**: Task 11 adds writer-input storage to social pipeline before the benchmark CLI is built. Without this, benchmark can replay newsletter only.

## Risks and mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Blended-donor extraction produces generic mush | High | Task 9 sanity check + fallback to single-donor (Packy McCormick) extraction |
| Writer inputs not stored historically — benchmark can only replay newsletter at first | Medium | Task 11 stores inputs going forward; benchmark falls back to newsletter-only on first run, social writers gain replay coverage after 5 days |
| Slack approval gate adds latency to daily run | Medium | 30-min auto-timeout per writer (Task 16 default). Pipeline never blocks indefinitely. |
| `voice-register.md` becomes stale | Low | Learning loop (Phase 6) auto-updates it. Re-run extraction (Task 8) when donor list changes. |
| Task 12 benchmark CLI grows beyond Large | Medium | Pre-emptively split into 12a (newsletter replay) and 12b (social replay) if it exceeds ~250 lines |
| Hard-fails scrubber over-corrects and breaks meaning | Medium | Task 4 scrubber returns `(text, fixes)` so all changes are auditable; tests cover golden inputs |
| Approval gate snapshot recording fails silently and we lose learning data | High | Task 15 wraps snapshot recording in try/except + logs WARNING; daily eval (Task 22) shows snapshot count so missing data is visible |
