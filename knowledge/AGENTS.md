# MindPattern Knowledge Base Schema

Adapted from Andrej Karpathy's LLM Wiki architecture. Unlike the original pattern which compiles from conversations alone, this system has **two input sources**: a daily AI research pipeline (13 agents, ~140 findings/day) AND interactive Claude Code conversations. Both feed the same wiki.

## The Compiler Analogy

```
daily/              = source code    (research findings - 13 agents, immutable)
conversations/      = source code    (your Claude Code sessions - immutable)
LLM                 = compiler       (extracts and organizes knowledge)
knowledge/          = executable     (structured, queryable knowledge base)
lint                = test suite     (health checks for consistency)
queries             = runtime        (using the knowledge)
```

You don't manually organize your knowledge. The pipeline runs at 7 AM, agents research 8 sources, MIRROR writes daily logs, and the compiler turns findings into wiki articles. When you have conversations in Claude Code, those get captured too. The LLM handles all the synthesis, cross-referencing, and maintenance.

## Architecture

### Layer 1: Research Daily Logs (Immutable Source)

Generated by MIRROR after each pipeline run. These are the "raw sources" for research knowledge. Append-only, never edited after the fact.

```
data/ramsay/mindpattern/daily/
├── 2026-02-12.md
├── 2026-02-13.md
├── ...
└── 2026-04-06.md     (51 files, growing daily)
```

Each file follows this format (generated by MIRROR, do NOT modify):

```markdown
---
type: daily
date: 2026-04-06
tags: daily, github-pulse, arxiv, news, ...
agents_run: 13
findings_count: 140
posts: 0
---

# Daily Activity Log - 2026-04-06

## Agents Run

### agents-researcher

Transcript: [[agents/agents-researcher/2026-04-06]]

**Findings:**
- Finding Title (importance) - [[topics/agents]]
  - Source: [[sources/source-domain]]
  - Summary of what was found...

### hn-researcher
...
```

### Layer 2: Conversation Logs (Immutable Source)

Each file is append-only. Multiple sessions per day are appended with timestamps:

The compiled knowledge base. The LLM owns this directory entirely. Humans read it; the LLM writes and maintains all of it.


## Structural Files

### knowledge/index.md — Master Catalog

A table listing every knowledge article. This is the primary retrieval mechanism — the LLM reads this FIRST when answering any query, then selects relevant articles to read in full.

At personal scale (50-500 articles), the LLM reading a structured index outperforms cosine similarity. The LLM understands what you're really asking; embeddings just find similar words.

Format:

```markdown
# Knowledge Base Index

## Concepts

| Article | Summary | Sources | Updated | Findings |
|---------|---------|---------|---------|----------|
| [[concepts/mcp-security]] | MCP protocol attack surfaces and mitigations | daily/2026-04-05, daily/2026-04-06 | 2026-04-06 | 15 |
| [[concepts/ai-coding-tools]] | AI coding assistant landscape and benchmarks | daily/2026-04-06, conversations/2026-04-06 | 2026-04-06 | 8 |

## Connections

| Article | Between | Updated |
|---------|---------|---------|
| [[connections/mcp-and-supply-chain]] | MCP Security ↔ Supply Chain Attacks | 2026-04-06 |

## Q&A

| Article | Question | Filed |
|---------|----------|-------|
| [[qa/auth-redirect-handling]] | How do I handle auth redirects? | 2026-04-06 |
```

### knowledge/log.md — Build Log

Append-only chronological record of every compile, query, and lint operation. Parseable with grep.

```markdown
# Build Log

## [2026-04-06T14:30:00Z] compile | daily/2026-04-06.md
- Source: daily/2026-04-06.md (140 findings)
- Articles created: [[concepts/mcp-security]], [[concepts/ai-coding-tools]]
- Articles updated: [[concepts/supply-chain-attacks]] (+3 findings)
- Connections: [[connections/mcp-and-supply-chain]]

## [2026-04-06T20:00:00Z] compile | conversations/2026-04-06.md
- Source: conversations/2026-04-06.md (3 sessions)
- Articles updated: [[concepts/knowledge-compiler-architecture]]
- New: [[concepts/obsidian-graph-patterns]]

## [2026-04-07T09:00:00Z] query | "What MCP vulnerabilities have been found?"
- Consulted: [[concepts/mcp-security]], [[concepts/supply-chain-attacks]]
- Filed to: [[qa/mcp-vulnerability-summary]]
```

## Article Formats

### Concept Articles (`knowledge/concepts/`)

One article per atomic piece of knowledge. These synthesize across multiple findings and conversations. A concept page about "MCP Security" pulls from 15+ findings across multiple days and possibly conversations where you discussed the topic.

```markdown
---
title: "Concept Name"
aliases: [alternate-name, abbreviation]
tags: [domain, topic]
sources:
  - "daily/2026-04-05"
  - "daily/2026-04-06"
  - "conversations/2026-04-06"
created: 2026-04-05
updated: 2026-04-06
confidence: high
finding_count: 15
---

# Concept Name

2-3 paragraph synthesis. Present tense. Connect dots between findings.
Be specific with names, numbers, dates. This is an encyclopedia entry,
not a news article.

## Key Points

- Insight 1 with specific evidence
- Insight 2 with specific evidence
- Insight 3 with specific evidence
(minimum 3, maximum 7)

## Details

Deeper analysis. Technical specifics, tradeoffs, implications.
Minimum 2 paragraphs. Use [[wikilinks]] to reference related concepts.

## Evidence

- [[daily/2026-04-06|Finding title from research]]
- [[daily/2026-04-05|Another finding]]
- [[conversations/2026-04-06|From debugging session]]
(cite every source used)

## Related

- [[concepts/related-concept-1]]
- [[concepts/related-concept-2]]
- [[connections/this-and-that]]
```

### Connection Articles (`knowledge/connections/`)

Cross-cutting synthesis linking 2+ concepts. Created when a conversation or set of findings reveals a non-obvious relationship. Don't create connections for obvious pairs — only when the relationship reveals something that wouldn't be apparent from reading either concept alone.

```markdown
---
title: "Connection: X and Y"
connects:
  - "concepts/concept-x"
  - "concepts/concept-y"
sources:
  - "daily/2026-04-06"
created: 2026-04-06
updated: 2026-04-06
---

# Connection: X and Y

## The Connection

What links these concepts. 1-2 paragraphs.

## Key Insight

The non-obvious relationship discovered.

## Evidence

- [[daily/2026-04-06|Finding that reveals the connection]]

## Related

- [[concepts/concept-x]]
- [[concepts/concept-y]]
```

### Q&A Articles (`knowledge/qa/`)

Filed answers from queries. Every complex question answered can be permanently stored, making future queries smarter. This is the compounding loop — every question makes the KB smarter.

```markdown
---
title: "Q: Original Question"
question: "The exact question asked"
consulted:
  - "concepts/article-1"
  - "concepts/article-2"
filed: 2026-04-06
---

# Q: Original Question

## Answer

The synthesized answer with [[wikilinks]] to sources.

## Sources Consulted

- [[concepts/article-1]] - Relevant because...
- [[concepts/article-2]] - Provided context on...

## Follow-Up Questions

- What about edge case X?
- How does this change if Y?
```

## Core Operations

### 1. Compile (daily/ + conversations/ → knowledge/)

**For Research Findings** (from `daily/*.md`):

1. Read today's daily log
2. Read `knowledge/index.md` to understand current knowledge state
3. Read existing articles that may need updating
4. For each thematic cluster of findings (3-8 clusters typical):
   - If an existing concept article covers this topic: **UPDATE** it with new information, add the daily log as a source, increment finding_count
   - If it's a new topic: **CREATE** a new `concepts/` article
   - If the findings reveal a non-obvious connection between 2+ existing concepts: **CREATE** a `connections/` article
5. **UPDATE** `knowledge/index.md` with new/modified entries
6. **APPEND** to `knowledge/log.md`

**For Conversations** (from `knowledge/conversations/*.md`):

1. Read the conversation log and extract: decisions made, lessons learned, architectural insights, debugging discoveries
2. For each insight, check if it belongs to an existing concept
3. If exists: merge the insight into the concept's Details section, add conversation as source
4. If new: create a new concept article (conversations can introduce concepts that research hasn't covered — project-specific patterns, codebase knowledge, personal workflow insights)
5. Update index and log

**Important guidelines:**

- A single daily log may touch 10-15 knowledge articles
- Prefer updating existing articles over creating near-duplicates
- Use Obsidian-style `[[wikilinks]]` — no `.md` extension
- Write in encyclopedia style — factual, concise, self-contained
- Every article must have YAML frontmatter
- Every article must link back to its source daily logs and/or conversations
- Summaries must be synthesis, not just listing findings

### 2. Query (Ask the Knowledge Base)

1. Read `knowledge/index.md` (the master catalog)
2. Based on the question, identify 3-10 relevant articles from the index
3. Read those articles in full
4. Synthesize an answer with `[[wikilink]]` citations
5. If `--file-back` is specified: create a `knowledge/qa/` article and update index.md and log.md

Why this works without RAG: at personal KB scale (50-500 articles), the LLM reading a structured index outperforms cosine similarity. The LLM understands what the question is really asking and selects pages accordingly.

### 3. Lint (Health Checks)

Seven checks, run periodically:

| Check | Type | Catches |
|-------|------|---------|
| Broken links | Structural | `[[wikilinks]]` pointing to non-existent articles |
| Orphan pages | Structural | Articles with zero inbound links |
| Orphan sources | Structural | Daily logs not yet compiled |
| Stale articles | Structural | Source daily log changed since compilation |
| Missing backlinks | Structural | A links to B but B doesn't link back |
| Sparse articles | Structural | Under 200 words |
| Contradictions | LLM | Conflicting claims across articles (requires judgment) |

Output: a markdown report in `reports/` with severity levels (error, warning, suggestion).

## Hook System (Automatic Capture)

Hooks fire automatically in Claude Code sessions within this project.

### .claude/settings.json

```json
{
  "hooks": {
    "SessionStart": [{ "matcher": "", "hooks": [{ "type": "command", "command": "python3 hooks/session_start.py", "timeout": 15000 }] }],
    "SessionEnd": [{ "matcher": "", "hooks": [{ "type": "command", "command": "python3 hooks/session_end.py", "timeout": 10000 }] }],
    "PreCompact": [{ "matcher": "", "hooks": [{ "type": "command", "command": "python3 hooks/pre_compact.py", "timeout": 10000 }] }]
  }
}
```

### Hook Details

**session_start.py** (SessionStart)
- Pure local I/O, no API calls, runs in under 1 second
- Reads `knowledge/index.md` and today's daily research log
- Outputs JSON: `{"additionalContext": "..."}`
- Claude sees the knowledge base index at the start of every session
- Max context: 20,000 characters

**session_end.py** (SessionEnd)
- Reads hook input from stdin (JSON with session_id, transcript_path, cwd)
- Extracts last 30 turns of conversation context (max 15,000 chars)
- Spawns `flush.py` as a fully detached background process
- Deduplication: checks `last-flush.json`, skips if same session flushed within 60s
- Recursion guard: exits immediately if `CLAUDE_INVOKED_BY` env var is set
- Minimum threshold: 2 user turns (skips trivial sessions)

**pre_compact.py** (PreCompact)
- Same architecture as session_end.py
- Fires before Claude Code auto-compacts the context window
- Higher minimum threshold: 5 user turns
- Critical for long sessions: captures context before summarization discards it

**Why both PreCompact and SessionEnd?** Long sessions trigger multiple auto-compactions. Without PreCompact, intermediate context is lost to summarization before SessionEnd fires.

### Background Flush Process (flush.py)

Spawned by both hooks as a fully detached background process:
- Mac/Linux: `start_new_session=True`
- Windows: `CREATE_NO_WINDOW` flag

What flush.py does:
1. Sets `CLAUDE_INVOKED_BY=flush` env var (prevents recursive hook firing)
2. Reads the pre-extracted conversation context from temp file
3. Skips if empty or if same session was flushed within 60 seconds
4. Calls `claude -p` with Sonnet to extract what's worth saving
5. Claude returns structured summary: Context, Key Exchanges, Decisions, Lessons, Action Items
6. Appends result to `knowledge/conversations/YYYY-MM-DD.md`
7. Cleans up temp context file
8. Logs all activity to `knowledge/state/flush.log`

## Compile Process (compile.py)

Uses `claude -p` via the orchestrator's existing `run_claude_prompt()`:

1. Fetches recent findings from SQLite (`memory.db`)
2. Clusters findings by embedding similarity (BAAI/bge-small-en-v1.5, 384-dim)
3. Matches clusters to existing concept pages
4. For each cluster, calls Claude to synthesize into a concept article
5. Writes concept pages to `knowledge/concepts/`
6. Regenerates `knowledge/index.md`
7. Appends to `knowledge/log.md`

**Incremental**: tracks what's been compiled in `knowledge/state/state.json`. Only recompiles when new findings arrive.

**CLI:**
```bash
python3 -m knowledge.compile                           # compile today
python3 -m knowledge.compile --date 2026-04-06         # specific date
python3 -m knowledge.compile --lookback 7              # last 7 days
python3 -m knowledge.compile --db data/ramsay/memory.db --vault data/ramsay/mindpattern
```

## State Tracking

`knowledge/state/state.json` tracks:
- `ingested` — map of daily log filenames to SHA-256 hashes and compilation timestamps
- `last_compile` — timestamp of most recent compilation
- `last_lint` — timestamp of most recent lint
- `concepts_count` — total concept articles
- `connections_count` — total connection articles

`knowledge/state/last-flush.json` tracks flush deduplication (session_id + timestamp).

Both are gitignored.

## Conventions

- **Wikilinks**: Obsidian-style `[[path/to/article]]` or `[[path/to/article|Display Text]]`, no `.md` extension
- **Writing style**: Encyclopedia-style, factual, present tense, specific names/numbers/dates
- **Dates**: ISO 8601 (`YYYY-MM-DD` for dates, full ISO for timestamps in log.md)
- **File naming**: lowercase, hyphens for spaces (e.g., `mcp-security-vulnerabilities.md`)
- **Frontmatter**: Every article must have YAML frontmatter with at minimum: title, sources, created, updated
- **Sources**: Always link back to the daily log(s) and/or conversation(s) that contributed
- **Minimum quality**: 200+ words per concept, 3+ key points, 2+ wikilinks, 2+ paragraphs in details
- **No duplicates**: Prefer updating existing articles over creating near-duplicates

## What NOT to Compile

- Temporary debugging context (one-off errors that were immediately fixed)
- Code snippets (save the decision/pattern, not the code)
- Pipeline operational details (run times, costs, error counts)
- Anything already in `soul.md`, `user.md`, or `voice.md`
- Findings that are just noise (low importance + no engagement + no recurrence)
- Greetings, confirmations, routine tool calls

## Safety Contract

**The daily pipeline (7 AM run) must NEVER be affected by the knowledge system.**

1. No changes to `runner.py`, `mirror.py`, or `memory/db.py`
2. MIRROR never touches `knowledge/` subdirectory
3. All new code lives in `knowledge/` and `hooks/` — pipeline doesn't import from either
4. Hooks check `MINDPATTERN_AGENT` env var and skip pipeline agents
5. `knowledge/` directory is outside MIRROR scope
6. **Test**: Delete `data/ramsay/mindpattern/knowledge/` → pipeline runs with zero errors

## Obsidian Integration

The knowledge base is pure markdown with `[[wikilinks]]` — works natively in Obsidian.

- Point vault at project root to see everything (research + codebase + wiki)
- Graph view shows concepts connected to daily logs, topics, and sources
- `knowledge/` nodes bridge the research vault and conversation knowledge
- Color groups in graph settings distinguish: daily (orange), topics (blue), sources (grey), agents (purple), knowledge (green), codebase docs (cyan)

## Scaling

At ~500+ concept articles or ~2M+ tokens, the index becomes too large for context. At that point, add hybrid search (BM25 + semantic) via `qmd` as a retrieval layer. qmd has both CLI and MCP server modes. For now, the index is sufficient.

## The Compounding Loop

```
Daily pipeline runs (7 AM)
    ↓
MIRROR: regenerates vault snapshot (raw layer)
    ↓
COMPILE: reads new findings → updates/creates concept pages
    ↓
INDEX: regenerated so next session has full catalog
    ↓
Next Claude Code session: SessionStart loads index
    ↓
You ask questions → LLM searches knowledge/ → synthesized answer
    ↓
Good answers filed back as Q&A articles → index updated
    ↓
SESSION CAPTURE: conversation → flush → conversations/ log
    ↓
Next COMPILE: conversation insights promoted into concepts
    ↓
Knowledge compounds. Repeat.
```
